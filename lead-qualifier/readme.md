# Bonsai Lead Qualifier Use Case Example

[How to setup this project?](setup.md)

## Overview

### 1. High-Level Goal

The system is a voice-first AI sales-qualification pipeline that acts as the front door for inbound leads at your company. Its job is threefold:

1. **Qualify** — gather BANT signal (Budget, Authority, Need, Timeline) through natural conversation
2. **Gate** — disqualify bad-fit leads gracefully, deflect guardrail situations appropriately
3. **Book** — schedule a 30-minute discovery call on a live Google Calendar and hand off to a human

The full loop is closed inside the system: the AI collects data, scores the lead, and books the meeting — without any human in the loop until the call itself.

---

### 2. Conversation Flow (Two Stages)

```
[Inbound Contact]
       │
       ▼
┌─────────────────────────────────┐
│  STAGE 1: Lead Qualifier v2     │  ← Alex qualifies the prospect via BANT
│  Transformers: Data Gathering   │
│               Director Whisper  │
└──────────────┬──────────────────┘
               │
        ┌──────┴──────────────┐
        │                     │
  is_disqualified        is_fully_qualified
  ────────────────        ─────────────────
  Generate polite         Summarize, transition
  rejection → END         go_to_stage → Stage 2
                                │
                                ▼
               ┌─────────────────────────────────────┐
               │  STAGE 2: Book Call                 │  ← Alex books the meeting
               │  Transformer: Director Whisper2     │
               │  on_enter: check_calendar auto-call │
               └──────────────────────────────────────┘
                    │
        Two-step confirmation flow:
        1. User proposes/accepts time
        2. Alex confirms: date + time + timezone
        3. User says "Yes" → schedule_call → END
```

Stage transition is fully automatic: when the `Director Whisper` transformer sets `is_fully_qualified: true`, the `all_data_collected_auto_trigger` action fires (watches the variable) and executes `go_to_stage` + `modify_user_profile` (saves BANT_note to user profile for Stage 2 to read).

### 3. Constants (Project-level Configuration)

```json
{
  "timezone": "Europe/Warsaw CET",
  "work_start": "9:00",
  "work_end": "17:00",
  "calendar_id": "c_79...@group.calendar.google.com",
  "company_name": "Company Name",
  "assistant_name": "Alex",
  "company_description": "Voice AI Innovative Software House",
  "qualification_matrix": "...",   // Full BANT scoring rubric
  "company_talking_points": "..."  // Trust-building facts
}
```

Constants are injected into **every prompt layer** — agent persona, both stage prompts, both transformers, and the Guardrails classifier. `qualification_matrix` is especially pervasive: it appears in the stage prompt (for the assistant's internal reasoning), in the Data Gathering transformer (for extraction guidance), in the Director Whisper (for scoring), and in the Guardrails classifier (for economic context when deciding how to respond to free-work requests).

Constants can be changed to adjust assistant to your business.

### 4. Calendar Tools (Bidirectional Integration)

`check_calendar`
- **Direction:** System → Google Calendar (read)
- **Params:** none
- **How it's used:** Called automatically on `__on_enter` of Stage 2. Result is stored in `vars.calendar_data` (string). The stage prompt renders it directly into the "Blocked hours" section, giving Alex real-time visibility of existing bookings. If not yet downloaded, a `[SYSTEM CALENDAR IS NOT DOWNLOADED YET. FORBIDDEN TO OFFER ANY DATES.]` block fires.

`schedule_call`
- **Direction:** System → Google Calendar (write)
- **Params:**
  - `date_start` — ISO 8601 with timezone offset (e.g. `2026-03-25T10:00:00+01:00`)
  - `date_end` — exactly 30 minutes after `date_start`
  - `description` — one-sentence BANT summary auto-generated by the LLM
- **How it's used:** Triggered only when `await_confirmation === true` AND classifier confirms the user's message is a confirmation. After the tool call, `vars.meet_link` is populated. The prompt detects `meet_link` and enters a final override state: stop all scheduling, provide the link, close the conversation. The LLM is explicitly told **not to read the link aloud** (voice UX consideration).

The **two-step confirmation** protocol is enforced at three layers simultaneously:
1. Classifier trigger condition (`await_confirmation === true`)
2. Stage prompt rule ("Never book immediately — verify date + timezone first")
3. Director Whisper2 transformer monitoring and flagging deviations in `inner_reflect`

### 5. Transformers — Roles and Mechanics

**`Data Gathering`** (Stage 1 only)

A pure **extraction** transformer. Runs after every turn and silently updates six stage variables from the conversation:

| Field | BANT Dimension |
|---|---|
| `client_pain_points` | Need |
| `project_timeline` | Timing |
| `client_role` | Authority |
| `client_budget` | Budget |
| `client_industry` | Context |
| `company_size` | Context |

It receives the current context values (`{{{json context}}}`) and only updates fields that changed. These fields are then rendered live into the stage prompt (the "COLLECTED DATA" block), so the assistant always sees what has and hasn't been gathered yet. No scoring logic here — this is purely structural data extraction.

---

**`Director Whisper`**

Director Whisper is **strategic oversight** transformer that run before each response. They write to multiple variables simultaneously and act as a hidden meta-layer on top of the agent.

**Shared output fields across both versions:**

| Variable | Type | Visible to Agent | Purpose |
|---|---|---|---|
| `inner_reflect` | string | No (hidden) | The Director's uncensored private reasoning — brutal assessment of lead quality, jailbreak detection, scheduling status |
| `tone_of_voice` | string | Yes (injected into prompt) | Directorial cue for Alex's demeanor this turn — written by the transformer, read by the agent |
| `director_suggestion` | string | Yes (injected into prompt) | Tactical instruction for Alex's next reply — "whisper in the ear" |

**Stage 1 Director Whisper also writes:**

| Variable | Purpose |
|---|---|
| `is_fully_qualified` | Set to `true` when BANT signal is strong enough — triggers stage transition |
| `is_disqualified` | Set to `true` on clear no-go signals — triggers conversation end |
| `red_flags` | Accumulated risk notes (persisted across turns by re-writing each time) |

The `inner_reflect` field is explicitly instructed to be written **first** in the JSON output (`always as the first field`) because it influences the model's chain-of-thought for the remaining fields — inner reasoning before decisions.

The Director Whisper is also the **auto-termination engine**: the transformer prompt includes a "Conversation Pacing & Decision Gate" section that mandates setting `is_disqualified: true` if the conversation stalls (3+ evasive answers, no BANT emerging, circular topics, no decision-making access). The system is designed so the AI proactively closes unproductive conversations without waiting for the user.

### 6. Dynamic Prompt Injection Techniques

The stage prompts are heavily template-driven using Handlebars-style syntax. Key patterns:

### Director's Whisper block
```handlebars
{{#exists vars.director_suggestion}}
### 🧠 DIRECTOR'S WHISPER (BACKGROUND GUIDANCE)
**[🚨 INTERNAL NOTE FROM DIRECTOR]:** {{vars.director_suggestion}}
*Incorporate this organically. Do not mention the Director.*
{{else}}
*No current directions. Proceed naturally.*
{{/exists}}
```
Every response cycle: the transformer runs → writes `director_suggestion` → the prompt renders it as an internal directive Alex must follow without revealing its source.

### Tone of Voice block
```handlebars
{{#exists vars.tone_of_voice}}
### 🗣️ SAMPLE COPY
Current tone of voice guidelines provided by director: *{{vars.tone_of_voice}}*
{{/exists}}
```
The transformer decides Alex's demeanor for the current turn (e.g., warmer if the user expressed stress, more direct if they seem decisive) and injects it as a persona cue. This creates a **dynamic, per-turn tone adaptation** rather than a fixed persona.

### Qualification status override
```handlebars
{{#if vars.is_fully_qualified}}
*[SYSTEM OVERRIDE: Lead is FULLY QUALIFIED. Ignore all intents, skills, and guardrails...]*
{{else if vars.is_disqualified}}
*[SYSTEM OVERRIDE: Lead is DISQUALIFIED...]*
{{else}}
*(If any of the below actions are triggered, adapt your response immediately.)*
...per-action blocks...
{{/if}}
```
When the Director Whisper sets a qualification verdict, the entire lower half of the prompt switches mode — suppressing all action-based guidance and giving a single overriding instruction.

### Per-action guidance injection (Skills & Guardrails)
```handlebars
{{#if actions.skill_vague_scope_clarifier}}
**Skill: User described their project in vague terms.**
[4-archetype framework + 4 diagnostic angles injected here]
{{/if}}

{{#if actions.guardrail_compliance_trap}}
**Guardrail: Regulated data detected.**
[GDPR/HIPAA/PCI-DSS/AI Act knowledge injected here]
{{/if}}
```
Skills and guardrail responses are not hardcoded — they're injected **conditionally based on classifier output**. The classifier runs first, sets `actions.X`, then the stage prompt renders the corresponding knowledge block directly into the LLM's context for that turn. This keeps the prompt lean at baseline and enriches it exactly when needed.

### Calendar state signals
```handlebars
{{#exists vars.calendar_data}}
{{else}}
*[SYSTEM CALENDAR IS NOT DOWNLOADED YET. FORBIDDEN TO OFFER ANY DATES.]*
{{/exists}}

{{#if vars.meet_link}}
*[SYSTEM OVERRIDE: Meeting is BOOKED. Provide the link, close permanently.]*
{{/if}}
```

### Collected data transparency
Both stage prompts render all currently known lead data inline — BANT fields, name, email, company — using `{{#exists}}` guards that show "❓ Unknown" for missing pieces. The agent sees its own state of knowledge each turn and naturally steers conversation toward the gaps.

### 7. Guardrail System — Classification Architecture

A dedicated `Guardrails` classifier (`clas_1771403097097_qw4grnp`) runs in **parallel** with the main `Intents&Events` classifier. Each user message is classified by both simultaneously.

**Escalation mechanic:** The Guardrails classifier receives `userProfile.guardrail_counter`. Once it reaches ≥ 3:

```
{{#gte userProfile.guardrail_counter 3}}
**🚨 SPECIAL INSTRUCTION:**
User repeatedly activated guardrails. You can only choose either
guardrail_safe (if fine) or guardrail_extremely_unsafe (to end conversation).
{{/gte}}
```

After 3 violations, the classifier locks into binary mode — no more nuanced handling, only safe pass or hard termination.

**10 guardrail types by consequence:**

| Category | Guardrail | Consequence |
|---|---|---|
| Hard terminates | `guardrail_extreme_unsafe` (illegal/NSFW/self-harm/hate) | `end_conversation` |
| Hard terminates | `guardrail_body_image_medical` | `end_conversation` |
| Soft + profile flag | `guardrail_jailbreak` | `generate_response` + `modify_user_profile` |
| Soft + profile flag | `guardrail_toxic` | `generate_response` + `modify_user_profile` |
| Soft + profile flag | `guardrail_free_work` | `generate_response` + `modify_user_profile` |
| Soft + profile flag | `guardrail_competitor_compare` | `generate_response` + `modify_user_profile` |
| Soft + profile flag | `guardrail_ideology_politics` | `generate_response` + `modify_user_profile` |
| Soft only | `guardrail_out_of_scope` | `generate_response` |
| Soft only | `guardrail_compliance_trap` | `generate_response` (with regulatory knowledge injected) |
| Pass-through | `guardrail_safe` | `generate_response` |

Guardrails that set `modify_user_profile` increment `guardrail_counter`, feeding the escalation mechanic above.
